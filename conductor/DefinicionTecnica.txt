DOCUMENTO DE DEFINICIÓN TÉCNICA: PROYECTO TFG

Título del Proyecto:
Plataforma Integral de Gestión Académica Multi-Sede con Auditoría Forense y Motor de Evaluación Híbrido

Alumno: [Tu Nombre] Tecnología: Java Spring Boot + PostgreSQL
1. Resumen Ejecutivo

Desarrollo de un sistema ERP educativo diseñado para garantizar la integridad del dato en entornos complejos. La plataforma resuelve la problemática de la gestión multi-sede (profesores compartidos entre centros) y ofrece un motor de evaluación flexible capaz de adaptarse a distintos criterios pedagógicos (ponderación por criterios, media por RAs, desglose teoría/práctica) sin alterar la estructura de la base de datos, gracias al uso de tipos híbridos (JSONB).

El sistema asegura la trazabilidad legal de las calificaciones mediante un módulo de auditoria forense a nivel de base de datos, inmutable desde la capa de aplicación.
2. Stack Tecnológico (Definitivo)

    Backend: Java 21 con Spring Boot 3.2.

        Seguridad: Spring Security 6 (RBAC estándar).

        ORM: Spring Data JPA + Hibernate (con hypersistence-utils para manejo de JSON).

    Frontend: Thymeleaf (Renderizado en servidor) + Bootstrap 5.

    Base de Datos: PostgreSQL 16.

        Características clave: Triggers PL/pgSQL, tipos JSONB, índices parciales.

    Infraestructura: Docker Compose (Contenedores para App, BD y pgAdmin).

3. Pilares de Arquitectura (La "Chicha" del TFG)
A. Gestión de Identidad y Roles (N:M)

    Diseño: Implementación de la interfaz UserDetails de Spring Security sobre un modelo relacional Muchos-a-Muchos (usuarios_roles).

    Ventaja: Permite perfiles híbridos reales (ej: un Jefe de Estudios que es ROLE_ADMIN y ROLE_PROFESOR simultáneamente) sin duplicidad de cuentas.

B. Arquitectura "Cortafuegos" (Defensa en Profundidad)

    Diseño: Desnormalización controlada del campo centro_id en tablas operativas críticas (imparticiones, matriculas).

    Ventaja: Garantiza que, incluso ante un error lógico en los JOINs de la aplicación, los datos de la "Sede Norte" están físicamente aislados de la "Sede Sur", permitiendo futuras políticas de seguridad a nivel de fila (RLS).

C. Motor de Evaluación Híbrido (JSONB)

    Diseño: Uso de columnas JSONB (configuracion_evaluacion) para definir las reglas de cálculo de notas.

    Funcionalidad: Soporta las casuísticas complejas exigidas:

        Cálculo por suma de Criterios vs. Media de RAs.

        Pesos variables entre Teoría y Práctica.

        Gestión de Recuperaciones (guardando la nota original en el histórico pero priorizando la nueva en el cálculo).
