Para asegurar una entrega profesional, se ha disenado un flujo de automatizacion en GitHub Actions dividido en cuatro fases criticas:

\begin{description}
    \item[Validacion y Estilo (Linting)] Audita cada linea de codigo. Utiliza Spotless para Java y Hadolint para Dockerfiles.
    \item[Construccion en Contenedor (Build)] La aplicacion se compila dentro de una imagen Docker aislada, garantizando paridad entre entornos.
    \item[Pruebas Reales (Test \& Quality Gate)] Se levanta una base de datos PostgreSQL real y se ejecutan las pruebas. El despliegue se bloquea si la cobertura de codigo (JaCoCo) es inferior al 80\%.
    \item[Purga y Promocion (Purge Policy)] Fase final que elimina herramientas de soporte y documentacion tecnica interna, creando una version de produccion pura en la rama final.
\end{description}
